1. Project Overview

WebPlot AutoDigitizer solves a common research problem: you have a scatterplot image from a published paper and need the underlying data as numbers. This tool lets you upload the image, confirm the axis calibration via interactive handles, and receive all plotted points as a downloadable CSV file.

Step 1: Upload

The user uploads a scatterplot image (PNG, JPG, or WEBP). The backend validates it (max 10MB, max 4000×4000 pixels), converts it to RGB, and stores it with a unique ID.

Step 2: Axis Detection

The system auto-detects the plot region using Canny edge detection and Hough line transform. It identifies horizontal and vertical axis lines, computes the plot bounding box, and optionally uses OCR (pytesseract) to read tick labels and determine the data range.

Step 3: User Calibration

The detected axes are shown as a canvas overlay with draggable blue handles at the axis endpoints. The user adjusts these handles to precisely align with the actual axis lines and enters the min/max data values for each axis. This step is critical — auto-detection is a starting suggestion, but the user's corrections are the source of truth.

Step 4: Digitization

The confirmed calibration is sent to the backend, which runs the HybridDigitizer. This combines blob detection (fast, ~1ms) with template matching (thorough, handles all marker shapes) and uses an agreement-based confidence comparison to select the best result. Each detected point gets data coordinates computed via the calibration transform.

Step 5: Export

Results are displayed as red circles overlaid on the original image, plus a data table showing X, Y, and confidence for each point. The user can copy the CSV to clipboard or download it as a file.

4. Digitization Methods & What We Learned

We implemented three detection methods and compared them empirically across 55 synthetic test plots spanning different marker shapes, sizes, opacities, backgrounds, and grid settings.

Method A: Blob Detection

How it works

Converts the plot region to grayscale, applies adaptive thresholding relative to the background median, then uses morphological operations and contour detection to find marker blobs. Computes centroids via image moments and derives a circularity-based confidence score.

Strengths

  • Extremely fast (~1ms per image)
  • High accuracy on circles, squares, and diamonds (95%+ detection)
  • Simple, robust to varying marker sizes

Weaknesses

  • Fails completely on triangle markers (0% — triangles don't form solid blobs after thresholding)
  • Struggles with X-shaped markers (thin cross shapes)
  • Confused by grid line intersections

Mean: 85.4% Speed: ~1ms

Method B: Template Matching

How it works

Generates a library of marker templates (circle, square, triangle, X, diamond) at multiple scales (7, 11, 15, 21 pixels). Runs normalized cross-correlation against the plot region for each template/scale combination, picks peaks above a threshold (0.4), applies non-maximum suppression, and selects the template/scale combo with the highest average score.

Strengths

  • Handles all marker shapes including triangles
  • Catches adversarial cases that blob detection misses
  • Multi-scale approach handles varying marker sizes

Weaknesses

  • Slow on grid plots (300–4000ms due to many false correlations)
  • More false positives overall
  • Lower precision on standard circular markers than blob detection

Mean: 78.5% Speed: 300-4000ms

Method C: Hybrid (Production)

How it works

Always runs both blob and template detection, then makes an intelligent selection:

  1. If one method finds no points, use the other
  2. Compute agreement: what fraction of blob points have a nearby template match (within 2% of axis range)?
  3. If agreement ≥50%: methods agree on locations — pick whichever has higher average confidence
  4. If agreement <50%: methods disagree — prefer template (handles non-standard markers better) unless blob confidence is significantly higher
  5. Fallback: merge both point sets with deduplication

Why this works

Blob detection is fast and precise on standard markers. Template matching is slow but comprehensive. By running both and comparing agreement/confidence, the hybrid gets the best of both worlds: it uses blob's precision when it's reliable, and template's coverage when the markers are unusual.

Mean: 91.9% Median: 95%

5. Test Harness & Evaluation

Philosophy: Test First, Always

The test harness was built before any digitization code. This was a non-negotiable design principle — we needed to measure accuracy objectively before and after every change. No "it looks right" testing.

Synthetic Plot Generator

The generator (tests/generate_plots.py) creates matplotlib scatterplots with known ground-truth coordinates, parameterized across:

  • Point count: 5, 20, 50, 200
  • Marker shapes: circle, square, triangle, X, diamond
  • Marker sizes: small, medium, large
  • Opacity: 1.0, 0.7, 0.4
  • Axis ranges: integers, floats, negative values
  • Grid lines: on/off
  • Background: white, light gray

Each plot is saved as a PNG with a companion JSON containing the exact x/y data values, axis ranges, and generation parameters.

Eval Scorer

The scorer (tests/eval_digitizer.py) uses a spatial KDTree approach:

  1. Normalize predicted and ground-truth points by the tolerance (default: 1% of axis range per axis)
  2. Build a KDTree from predicted points
  3. Query each ground-truth point for its nearest predicted neighbor
  4. A match requires distance ≤ √2 in normalized space (within tolerance on both axes)
  5. Report: matched %, mean error X/Y, max error, false positives, false negatives

6. Frontend UI

The frontend is a three-step wizard built with React 18 and TypeScript, using the Canvas API for interactive overlays:

Step 1: Image Upload

Drag-and-drop zone or file picker. Validates image type and shows a preview. Sends the file to /api/upload and receives an image ID.

Step 2: Axis Calibration

Canvas overlay shows the uploaded image with blue dashed axis lines and draggable circular handles. Four numeric inputs let the user enter X min/max and Y min/max data values.

Step 3: Results View

Red circles overlaid on the image at each detected point location, with opacity proportional to confidence. A scrollable data table shows point index, X, Y, and confidence %.

CSV Export

Two buttons: "Copy CSV" (copies comma-delimited rows to clipboard) and "Download CSV" (triggers a file download). Users can go back to adjust calibration or start over.

The UI is responsive and usable on tablet-sized screens. All async operations show loading states, and errors are displayed clearly in red.

Option 1: Docker (Recommended)

The fastest way to run the full application. The Docker image builds both the frontend and backend, serving everything on a single port.

docker-compose up

9. Key Design Decisions

Why Test-First?

Image processing is notoriously hard to evaluate by eye. A point detection algorithm might look "good enough" on one image but fail catastrophically on another. By building the synthetic test harness first (Phase 1, before any detection code), we ensured every method was evaluated on the same 55 plots with the same scoring criteria. This let us make data-driven decisions about which approach to ship.

Why Multiple Methods?

No single computer vision technique handles all marker shapes equally well. Blob detection is fast and precise for filled shapes but fails on triangles and crosses. Template matching handles everything but is slow and produces more false positives. By implementing both and comparing empirically, we discovered the hybrid approach that gets 91.9% — better than either alone.

Why User Calibration is King

Auto-detected axes are a convenience, not a requirement. Real-world plots have legends, annotations, insets, and non-standard layouts that confuse automated detection. The draggable fiducial handles let users correct any axis detection errors, making the system robust even when auto-detection completely fails.

Why CPU-Only?

The target deployment is cheap cloud hosting (Railway, Render, Fly.io) without GPU access. All image processing uses OpenCV and NumPy on CPU. The hybrid digitizer completes in under 4 seconds even in worst-case scenarios, well within the 10-second budget.

Why a Single Docker Port?

In production, the FastAPI backend serves the compiled React frontend as static files from the same port (8000). This eliminates CORS issues, simplifies deployment to a single container, and means users access everything at one URL.
