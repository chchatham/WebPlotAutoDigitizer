1. Project Overview

WebPlot AutoDigitizer solves a common research problem: you have a scatterplot image from a published paper and need the underlying data as numbers. This tool lets you upload the image, confirm the axis calibration via interactive handles, and receive all plotted points as a downloadable CSV file.

Step 1: Upload

The user uploads a scatterplot image (PNG, JPG, or WEBP). The backend validates it (max 10MB, max 4000×4000 pixels), converts it to RGB, and stores it with a unique ID.

Step 2: Axis Detection

The system auto-detects the plot region using a dual-strategy approach: background-color detection (for ggplot2/seaborn-style plots with gray backgrounds) combined with Canny edge detection and Hough line transform. It identifies horizontal and vertical axis lines, computes the plot bounding box using whichever method returns the larger area, and optionally uses OCR (pytesseract) to read tick labels and determine the data range.

Step 3: User Calibration

The detected axes are shown as a canvas overlay with independent draggable handles: blue handles for the X-axis (horizontal movement repositions the handle, vertical movement translates both handles together) and green handles for the Y-axis (vertical to reposition, horizontal to translate). The axis lines themselves are also draggable. An 8× magnified zoom panel tracks the active handle for pixel-precise placement. An orange detection bounding box defines where to search for points, and four numeric inputs let the user enter the data ranges. An optional "Expected number of points" field guides the clump decomposition algorithm.

If the user clicks "Adjust calibration" from the results screen, all handle positions, data ranges, and the bounding box are restored from the previous calibration — the system never re-runs axis detection and discards the user's work.

Step 4: Digitization

The confirmed calibration is sent to the backend, which runs the HybridDigitizer. This combines blob detection (fast, ~1ms) with template matching (thorough, handles all marker shapes) and shape-aware clump decomposition (for overlapping markers). It uses an agreement-based confidence comparison to select the best result, and each detected point gets data coordinates computed via the calibration transform. All numeric values are sanitized (NaN/Infinity converted to 0.0, numpy scalars cast to native Python float) before serialization.

Step 5: Export

Results are displayed as translucent red circles overlaid on the original image (max opacity ~0.225 so original markers show through), plus a data table showing X, Y, and confidence for each point. The user can copy the CSV to clipboard or download it as a file, then go back to adjust calibration or start over.

4. Digitization Methods

We implemented four detection methods and compared them empirically across 98 synthetic test plots spanning different marker shapes, sizes, opacities, backgrounds, grid settings, overlap levels, and fill styles.

Method A: Blob Detection

How it works

Converts the plot region to grayscale, applies adaptive thresholding relative to the background median, then uses morphological operations and contour detection to find marker blobs. Merged contours (area >1.3× median or elongation >1.4) are split via distance-transform watershed with progressive peak thresholds (0.35, 0.25, 0.18). Computes centroids via image moments and derives a circularity-based confidence score.

Strengths

  • Extremely fast (~1ms per image)
  • High accuracy on circles, squares, and diamonds (95%+ detection)
  • Built-in watershed splitting for overlapping markers

Weaknesses

  • Fails on triangle markers (0% — triangles don't form solid blobs after thresholding)
  • Struggles with X-shaped markers (thin cross shapes)
  • Confused by grid line intersections

Mean: 85.4% Speed: ~1ms

Method B: Template Matching

How it works

Generates a library of marker templates (circle, square, triangle, X, diamond) at multiple scales (7, 11, 15, 21 pixels). Runs normalized cross-correlation against the plot region for each template/scale combination, picks peaks above a threshold (0.4), applies non-maximum suppression, and selects the template/scale combo with the highest average score.

Strengths

  • Handles all marker shapes including triangles
  • Catches adversarial cases that blob detection misses
  • Multi-scale approach handles varying marker sizes

Weaknesses

  • Slow on grid plots (300–4000ms due to many false correlations)
  • More false positives overall
  • Lower precision on standard circular markers than blob detection

Mean: 78.5% Speed: 300-4000ms

Method C: Hybrid (Production Router)

How it works

The production digitizer combines all other methods via intelligent routing:

  1. Always runs blob detection first (fast baseline)
  2. Checks for significant clumping: >5% of contour area in merged blobs, or user provided an expected point count
  3. If clumps detected: runs ShapeAwareDetector (Method D) and compares — prefers shape-aware if it finds more points with reasonable confidence
  4. Otherwise: runs template matching and computes agreement with blob results
  5. If agreement ≥50%: pick whichever method has higher average confidence
  6. If agreement <50%: prefer template unless blob confidence is significantly higher
  7. Fallback: merge both point sets with deduplication

Mean: 91.9% Median: 95%

5. Shape-Aware Clump Decomposition

Real-world scatterplots frequently have overlapping or touching markers. A naive blob detector sees merged markers as a single large contour and reports one point where there are two or more. Phase 14 added a four-stage pipeline to handle this.

Stage 1: Marker Profile Estimation

Before decomposing clumps, the system learns what a single marker looks like by analyzing isolated singletons in the image:

  • Finds all contours, filters by area (0.005%–2% of ROI), computes median area
  • Identifies singletons: contours within [0.4x, 1.8x] median area
  • Filters for isolation: removes contours with neighbors within 2.5x median radius (KDTree)
  • Computes profile: median radius, area, circularity, fill ratio
  • Hollow detection: fill_ratio < 0.7 means unfilled/hollow markers
  • Requires minimum 3 singletons; falls back to standard blob detection if insufficient

Output: MarkerProfile(mean_radius_px, std_radius_px, mean_area_px, is_hollow, edge_width_px, circularity, fill_ratio, n_singletons)

Stage 2: Clump Classification

Each contour is classified as a singleton or clump based on:

  • Area threshold: contour area > 1.15× (with hint) or 1.3× (without hint) the marker profile's mean area
  • Elongation: bounding box aspect ratio > 1.3 AND area > mean area (catches peanut-shaped 2-point overlaps)

Stage 3: Filled-Marker Decomposition

For filled markers, multiple strategies compete to find the best centroid placement:

  1. Erosion split: progressively erodes the clump mask until it separates into N components; most accurate centroids when it works
  2. Ellipse-based placement: fits an ellipse to the contour, distributes N points along the major axis at geometrically correct spacing (half_len = major/2 - radius), then refines each point by finding the local distance-transform maximum
  3. Distance-transform NMS: computes the distance transform, finds local maxima with non-maximum suppression at 0.6× radius spacing, tries multiple threshold fractions (0.1–0.5)
  4. Label-based decomposition: thresholds the distance transform at progressively lower values, labels connected components, computes their centroids
  5. Skeleton spacing: for large clumps (3+ points), distributes points evenly along the morphological skeleton

The strategy producing the count closest to the estimated N wins. The DT refinement step uses mutual exclusion: after placing a point, the distance transform is suppressed around it so the next point can't collapse to the same peak.

Clump recall: 100% at 2% tolerance (filled, ≤50% overlap)

Stage 4: Hollow-Marker Decomposition

For hollow/unfilled markers, Hough Circle Detection is preferred since the circular edges are well-defined:

  • Extracts a sub-ROI around the clump with 2× radius padding
  • Searches with radius tolerance ±max(4, 35% of radius) pixels
  • Sweeps 5 Hough parameter configurations (dp, minDist, Canny threshold, accumulator threshold)
  • Adaptive Gaussian blur kernel sized proportional to marker radius
  • Confidence based on how close detected radius matches the profile: conf = 0.6 + 0.3 × (1 - radius_diff)
  • Falls back to filled decomposition if Hough finds too few circles

Clump recall: 91–100% at 2% tolerance (hollow, ≤50% overlap)

User Point Count Hint

The frontend provides an optional "Expected number of points" input. When provided, it acts as a soft constraint (weight 0.3) that biases decomposition thresholds and always activates the ShapeAwareDetector regardless of clump area percentage. The hint never forces exact agreement — the algorithm degrades gracefully when given an incorrect count.

6. Calibration UI

The calibration screen is the most complex part of the frontend. It combines multiple interactive overlays on a single canvas:

Independent Axis Handles

  • X-axis (blue): two handles connected by a horizontal line. Horizontal drag repositions that handle's X coordinate; vertical drag translates both handles together (moves the X-axis line up/down).
  • Y-axis (green): two handles connected by a vertical line. Vertical drag repositions that handle's Y coordinate; horizontal drag translates both handles together (moves the Y-axis line left/right).
  • The axis lines themselves are also draggable for quick repositioning.

8× Precision Zoom Panel

A 200×200px panel that magnifies the area around the active handle at 8× zoom with crosshairs. Reads from the original-resolution image (not the scaled canvas) for pixel-precise alignment. Uses imageSmoothingEnabled = false for crisp magnification.

Detection Bounding Box

An orange dashed rectangle with 4 draggable corner handles defining WHERE to search for points. Initialized with 15% padding beyond the axis range, clamped to image dimensions. This is independent from the axis handles (which define HOW to map pixel coordinates to data coordinates).

7. Test Harness & Evaluation

Philosophy: Test First, Always

The test harness was built before any digitization code. This was a non-negotiable design principle — we needed to measure accuracy objectively before and after every change. No "it looks right" testing.

Synthetic Plot Generator

The generator (tests/generate_plots.py) creates matplotlib scatterplots with known ground-truth coordinates, parameterized across:

  • Point count: 5, 20, 50, 100, 200
  • Marker shapes: circle, square, triangle, X, diamond
  • Marker sizes: small (20), medium (40), large (80), xlarge (120)
  • Fill style: filled or unfilled (with configurable edge width)
  • Marker color: black, red, blue, or any matplotlib color
  • Opacity: 1.0, 0.7, 0.4
  • Overlaps: configurable overlap fraction (15–70%), pair count, triple count, isolated count
  • Axis ranges: integers, floats, negative values
  • Grid lines: on/off
  • Background: white, light gray, medium gray

Each plot is saved as a PNG with a companion JSON containing the exact x/y data values, axis ranges, and generation parameters.

Eval Scorer

The scorer (tests/eval_digitizer.py) uses a spatial KDTree approach with two matching modes:

  1. Normalize predicted and ground-truth points by the tolerance (default: 1% of axis range per axis)
  2. Build a KDTree from predicted points, query each ground-truth point for its nearest neighbor
  3. Standard matching: any prediction within tolerance counts as a match (multiple truths can match the same prediction)
  4. Unique matching: greedy assignment sorted by distance — each prediction matches at most one truth point (more realistic for clump evaluation)
  5. Report: matched %, mean error X/Y, max error, false positives, false negatives, clump recall, singleton recall, clump precision

8. Error Handling & Robustness

Several production bugs were discovered and fixed during deployment, leading to comprehensive error hardening:

Frontend

  • Error Boundary: wraps all app content; catches render crashes and shows a "Start over" recovery button instead of a blank page
  • Robust catch handlers: Promise .catch() uses e instanceof Error ? e.message : String(e ?? "Unknown error") — never assumes e has a .message property
  • Response validation: api.ts validates that digitize responses have a points array before returning

Backend

  • Digitize try/except: wraps digitizer.digitize() and returns 500 with a descriptive message on failure
  • Pixel range validation: rejects zero-width pixel ranges with 400 (prevents ZeroDivisionError)
  • NaN/Infinity sanitization: all numeric output is checked and converted to 0.0 if not finite
  • numpy scalar casting: sanitize() calls float(v) on all values to convert numpy.float32 to native Python float before FastAPI serialization

11. Key Design Decisions

Why Test-First?

Image processing is notoriously hard to evaluate by eye. A point detection algorithm might look "good enough" on one image but fail catastrophically on another. By building the synthetic test harness first (Phase 1, before any detection code), we ensured every method was evaluated on the same plots with the same scoring criteria. This let us make data-driven decisions about which approach to ship.

Why Multiple Methods?

No single computer vision technique handles all marker shapes equally well. Blob detection is fast and precise for filled shapes but fails on triangles and crosses. Template matching handles everything but is slow and produces more false positives. Shape-aware decomposition handles overlaps but requires singletons to calibrate. By implementing all four and routing intelligently, the hybrid gets the best of each.

Why User Calibration is King

Auto-detected axes are a convenience, not a requirement. Real-world plots have legends, annotations, insets, and non-standard layouts that confuse automated detection. The draggable fiducial handles let users correct any axis detection errors, and calibration state persists when the user goes back to adjust — their work is never lost.

Why CPU-Only?

The target deployment is cheap cloud hosting (Railway, Render, Fly.io) without GPU access. All image processing uses OpenCV and NumPy on CPU. The hybrid digitizer completes in under 4 seconds even in worst-case scenarios, well within the 10-second budget.

Why a Single Docker Port?

In production, the FastAPI backend serves the compiled React frontend as static files from the same port. This eliminates CORS issues, simplifies deployment to a single container, and means users access everything at one URL.

Why Unique Matching in Scoring?

Standard KDTree matching lets multiple ground-truth points match the same prediction, inflating recall when clumped points collapse to a single detection. Unique matching (greedy assignment) gives a realistic measure of how many individual points the detector resolves. This was critical for measuring clump decomposition quality.
